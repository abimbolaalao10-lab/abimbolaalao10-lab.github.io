Professional Self-Assessment

Why these three artifacts, together.

CS 499 Computer Science Capstone · Abimbola Alao

The three artifacts in this ePortfolio were not built to showcase isolated skills. They were chosen because, together, they tell an honest story.

Completing the Computer Science program at Southern New Hampshire University, and specifically the work of this capstone, has clarified both what I can build and how I want to build it. This self-assessment introduces the three artifacts in my ePortfolio a Java service management application, the same application enhanced with search and sort algorithms, and a Python machine learning web application enhanced with database persistence and explains how the process of enhancing them demonstrates the skills and values I bring to a software engineering role.

Throughout the program, my coursework spanned several areas that all converge in the work presented here: software development fundamentals in Java and Python, software testing and quality assurance, database design and SQL, full-stack web development with the MEAN stack, and independent project work building blockchain applications, decentralized auction platforms, and a research-grade phishing detection system using machine learning. Alongside coursework, I have pursued a self-directed cybersecurity learning path, building habits of thinking about software through an adversarial lens that show up directly in the enhancements documented in this ePortfolio.

01 Collaborating in a Team Environment

Although this capstone was completed independently, the artifacts themselves were built with collaboration in mind, and my broader project work has given me direct experience working with and for others. As the founder and sole smart contract developer of AstaVerse, a decentralized auction platform, I have written grant applications that required translating technical architecture into language a non-technical funding committee could evaluate. As a freelance developer, I have built features for the PEWITO PropTech platform based on UX strategy briefs developed by a separate design team, requiring me to interpret someone else's design intent faithfully in code.

The code review video that opens this ePortfolio is itself an exercise in communicating technical analysis to an audience in that case, an instructor evaluating my ability to critique my own past work honestly. The discipline of identifying weaknesses in code I had already submitted for a grade, rather than defending it, reflects the kind of intellectual honesty that effective technical collaboration requires.

02 Communicating with Stakeholders

Every artifact in this ePortfolio is accompanied by a written narrative that explains not just what was built, but why, and what trade-offs were considered along the way. Writing these narratives required adapting my communication style across audiences: the Javadoc comments embedded throughout the Java codebase are written for a developer audience and use precise technical vocabulary, while the narrative documents accompanying each milestone are written for an audience that includes both technical reviewers and, in this final ePortfolio, potential employers who may not have time to read every line of code.

This skill was sharpened through real-world work outside the classroom as well. Writing Stellar Community Fund grant applications for AstaVerse required me to describe the same technical architecture in different registers depending on the reviewer. Producing technical budget presentations for the PEWITO platform required translating engineering estimates into a format a business stakeholder could act on. This self-assessment is itself the clearest example: a single document meant to introduce my technical work to an audience that includes hiring managers, who need a coherent narrative more than a feature list.

03 Algorithmic Design and Software Engineering Practices

The Software Design and Engineering and Algorithms and Data Structures artifacts in this ePortfolio both built on the same CS 320 Java project demonstrate this outcome most directly. The original implementation stored Contact, Task, and Appointment records in a plain HashMap, providing fast key-based lookup but no way to search, sort, or reliably iterate the data in a predictable order.

Enhancing this artifact required two distinct kinds of design thinking. First, introducing service interfaces (IContactService, ITaskService, IAppointmentService) required reasoning about the system architecturally decoupling the contract a service promises to its callers from the specific way it is implemented. Second, replacing HashMap with LinkedHashMap and adding Stream-based search and Comparator-based sort methods required evaluating concrete algorithmic trade-offs: an O(n) linear scan was the right choice for search at this scale, while a secondary index would have added complexity without a justified benefit. Documenting that reasoning in Javadoc, rather than just implementing the change, is what distinguishes engineering from coding.

This same instinct to evaluate trade-offs explicitly rather than default to the most familiar solution shows up in my independent project work as well, particularly in the architecture decisions behind the AstaVerse Stellar integration, where I chose a polling-based cross-chain event listener specifically because of documented RPC limitations on Base Sepolia, rather than assuming a webhook-based design would work universally.

04 Innovative Techniques, Skills, and Tools

Across all three artifacts, I applied tools and techniques appropriate to each problem rather than reaching for a single familiar stack regardless of fit. The Java enhancements used interface-driven design, the Streams API, and the Comparator API idiomatic modern Java rather than the more verbose patterns common in introductory coursework. The Python phishing detector enhancement used SQLite specifically because it matched the scale and deployment context of a research prototype, rather than defaulting to a heavier client-server database that the application did not need.

This same judgment selecting the right tool for the actual constraints of a project, not the most impressive one runs through my independent work. Building a Telegram-controlled NFT minting bot required integrating python-telegram-bot, web3.py, and the Reservoir API to build a phase-eligibility engine that reads on-chain state directly, because relying on an off-chain database would have introduced a synchronization risk the application could not tolerate.

05 Security Mindset

The Databases enhancement to the phishing email detector is the clearest demonstration of this outcome. The original application had no persistence layer at all; adding one required designing a schema and a set of data access functions with security as a first-order concern rather than an afterthought. Every database query in the enhanced application uses parameterized statements including the LIMIT clause in the prediction history query, a detail that is easy to overlook because it does not come from an HTML form, but is still a variable incorporated into SQL and therefore still a potential injection vector if handled carelessly.

Beyond the database layer, the decision to store only a 300-character snippet of each analyzed email, rather than the full text, reflects a privacy-by-design instinct: store the minimum data necessary for the application's purpose, because any data retained is data that can eventually be exposed. This security-first orientation is not limited to coursework. My self-directed cybersecurity learning path including a 14-week structured curriculum I built and packaged as a desktop learning tracker reflects a deliberate, ongoing investment in this outcome that extends well beyond what any single course requires.

Conclusion

The three artifacts in this ePortfolio were not built to showcase isolated skills. They were chosen because, together, they tell an honest story: a developer who can look critically at code written under deadline pressure in an earlier course, identify exactly where it falls short of professional standards, and close those gaps with documented, deliberate engineering decisions. That is the work of this capstone, and it is the kind of work I intend to keep doing as a backend software engineer building systems that are not just functional, but well-designed, secure, and built to be understood by the next person who has to maintain them.
