01 — SOFTWARE DESIGN & ENGINEERING

Contact, Task & Appointment Services

A Java service layer from CS 320, rebuilt around interfaces, logging, and consistent validation.

Java · JUnit 5

Three services, one consistent contract

Originally built for CS 320: Software Testing, Automation, and Quality Assurance
Original + enhanced code (.zip) Narrative (.docx)

The original implementation worked — tests passed — but had no service interfaces, no logging, an incomplete CRUD surface on AppointmentService, and inconsistent null-handling between modules. The enhancement closes each gap and documents why, not just what.

  • IContactService / ITaskService / IAppointmentService added — callers depend on a contract, not an implementation
  • java.util.logging added across all three services for a real audit trail
  • AppointmentService CRUD completed — date and description updates were missing entirely
  • Null-checking and error handling standardized into a shared requireX() pattern
  • Full Javadoc on every class, method, and field
  • Test suite expanded with boundary values, null inputs, and non-existent-ID cases
02 — ALGORITHMS & DATA STRUCTURES

Same codebase, rebuilt around how data moves

The storage layer gains order, search, and sort — each with its complexity documented in writing.

Java · Streams · Comparator

From key-lookup-only to a real query surface

Builds on the Milestone Two enhancement of the same CS 320 project
Original + enhanced code (.zip) Narrative (.docx)

A plain HashMap gives fast lookup by ID and nothing else — no predictable order, no way to search by name, no way to sort. The enhancement replaces it with a structure and a set of methods fit for how the data is actually used.

  • HashMap → LinkedHashMap — preserves insertion order at the same O(1) cost
  • getAllContacts() / getAllTasks() / getAllAppointments() — O(n) full retrieval
  • searchByName() / searchByDescription() — O(n) Stream filter, case-insensitive
  • sortByLastName() / sortByName() / sortByDate() — O(n log n) Comparator sort, original map never mutated
  • Two-level comparator (last name, then first name) for deterministic tie-breaking
  • Tests confirm sorting never disturbs the underlying insertion order
03 — DATABASES

Phishing Email Detector

A 99.74%-accuracy Random Forest classifier, rebuilt around a database it never had.

Python · Gradio · SQLite

From stateless prediction to an auditable system

Originally a postgraduate research project — no persistence layer at all
Original + enhanced code (.zip) Narrative (.docx)

The original app analyzed an email and forgot it the instant the result was shown — no history, no audit trail, no way for a user to flag a wrong prediction. For a security tool, that gap is not cosmetic. The enhancement adds a small, carefully scoped SQLite layer to close it.

  • predictions table — every analysis logged with timestamp, result, confidence, indicators
  • feedback table — foreign key to predictions, ON DELETE CASCADE
  • Every query parameterized — including the LIMIT clause, a detail that's easy to miss
  • PRAGMA foreign_keys = ON set per connection — SQLite disables this by default
  • Only a 300-character snippet of each email stored — privacy-by-design, not full content
  • New Submit Feedback and Prediction History tabs added to the Gradio UI

"The three artifacts here were not built to showcase isolated skills — they were chosen because together they tell one honest story."

Each artifact in this ePortfolio started as code that worked but fell short of a professional standard in a specific, identifiable way. The value I tried to demonstrate isn't that I can write code — it's that I can look critically at code I already submitted for a grade, name exactly where it falls short, and close that gap with a decision I can defend in writing: why LinkedHashMap instead of HashMap, why SQLite instead of a heavier database, why a linear scan is the right search strategy at this scale instead of a more "impressive" index structure.

That instinct — find the gap, justify the fix, document the trade-off — is the same one I bring to independent work outside the classroom: a decentralized auction platform on Base, a Telegram-controlled NFT minting bot with an on-chain eligibility engine, and a property-rental platform built for the Nigerian market. It's the kind of engineering I want to keep doing professionally.
